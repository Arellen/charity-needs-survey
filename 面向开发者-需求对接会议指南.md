# 公益小程序 · 需求对接会议指南

> 使用说明：本指南覆盖从会议准备到会后确认的完整流程。
> 你作为开发者与慈善机构对接时，照此流程执行即可。

---

## 一、整体思路：把开放式讨论变成"选择题 + 填空题"

慈善机构的人不懂技术，如果问"你们想要什么功能"，他们会要么回答不上来，要么提一堆不现实的需求。

**正确做法**：你先准备好选项，让他们做选择题；开放性问题只保留最关键的那几个。

工具组合：
1. **会前**：发《腾讯文档》给机构提前看
2. **会中**：用《腾讯问卷》逐项过，当场记录
3. **会后**：发《需求确认书》让机构签字

---

## 二、会议前：你需要做的

### 2.1 提前 3-5 天发给机构

发送内容：
```
【XX机构】您好，附件是我们准备的小程序功能介绍文档，
请贵机构相关负责人在会前通读一遍，了解大致功能范围。
会上我们会逐一确认具体需求，预计会议时间 1.5-2 小时。

文档链接：[腾讯文档链接]
```

### 2.2 你自己会前准备

- [ ] 通读机构的官网/公众号/现有项目，了解他们做什么领域
- [ ] 准备 3-5 个类似机构的小程序案例截图（如腾讯公益、水滴筹等），用于"你更喜欢哪种"的问题
- [ ] 打印一份《腾讯问卷》题目清单，会上逐题过
- [ ] 准备录音笔或手机录音（征得对方同意后使用）
- [ ] 建一个会议纪要文档模板

---

## 三、会议中：按模块逐项确认（1.5-2小时流程）

### 开场（5分钟）

**你说**：
> "今天主要是确认三件事：小程序要做哪些功能、你们有哪些现成的资料可以用、大概的时间预期。我们有准备好的功能清单，按清单逐项过，你们觉得合适的就保留，不需要的就去掉，不确定的先标记。这样效率最高。"

---

### 第一轮：确认功能范围（60-80分钟）

**方式**：打开腾讯文档，逐模块过。每个模块按以下节奏：

1. 你花 1 分钟讲清楚这个功能是什么（用类比，不要用术语）
2. 问机构："这个要不要？"
3. 如果要，追问关键细节（见下方提问清单）
4. 当场在问卷上记录答案

---

**模块 A：捐款模块 · 关键提问清单**

| 问题 | 为什么问这个 | 记录 |
|---|---|---|
| 你们有公众号吗？有没有认证？ | 公众号可以作为引流入口，小程序的推送也需要它 | |
| 现在捐款人怎么给你们捐钱？银行转账还是微信？ | 了解现有流程，小程序上线后怎么过渡 | |
| 需不需要分项目展示？项目分类有哪些？ | 决定首页的筛选维度 | |
| 捐款金额档位你们现在怎么设的？ | 决定默认档位 | |
| 发票是你们自己开还是委托基金会开？ | 决定发票申请功能做不做 | |
| 有没有合作商家/企业可以嵌入捐款入口？ | 决定场景化捐款入口要不要第一期做 | |
| 捐款人能不能看到自己的钱具体帮了谁？ | 决定反馈闭环的粒度（是项目级还是受助人级） | |

---

**模块 B：受助人模块 · 关键提问清单**

| 问题 | 为什么问这个 | 记录 |
|---|---|---|
| 你们现在受助人信息怎么记录的？Excel还是纸质？ | 了解数据迁移工作量 | |
| 表格里有哪些字段？给我看看模板 | 决定录入表单的字段设计 | |
| 谁去现场录入？是你们自己的员工还是合作机构的人？ | 决定要不要做合作机构权限管理 | |
| 审核流程走几级？谁有最终决定权？ | 决定审核流程设计 | |
| 证明材料现在存哪里？有没有电子版？ | 决定图片存储容量预估 | |
| 一个受助人会不会被多个项目帮扶？ | 决定数据关联是1对1还是多对多 | |
| 有没有老年人受助人不会用手机？怎么处理？ | 决定要不要做代操作功能 | |
| 数据要不要定期导出？导出给谁看？ | 决定Excel导出功能的字段和格式 | |

---

**模块 C：项目管理模块 · 关键提问清单**

| 问题 | 为什么问这个 | 记录 |
|---|---|---|
| 谁可以创建项目？需要审批吗？谁审批？ | 决定权限层级 | |
| 项目预算分几大类？ | 决定预算模板结构 | |
| 善款是全部给受助人还是可以提取管理费？比例多少？ | 决定资金计算逻辑 | |
| 项目执行期多长？一次性还是分阶段？ | 决定项目时间线设计 | |
| 项目结项要哪些材料？谁审核？ | 决定结项流程 | |
| 需不需要对外公开资金明细？公开到什么程度？ | 决定透明度的设计 | |

---

**模块 D：进阶功能 · 关键提问清单**

| 问题 | 为什么问这个 | 记录 |
|---|---|---|
| 有没有企业表示愿意配捐？或者你们有企业合作伙伴？ | 决定配捐功能要不要第一期做 | |
| 是否希望受助人付1分钱而不是完全免费？ | 决定尊严支付功能 | |
| 有没有合作的社区、村委会、其他NGO？ | 决定合作机构角色 | |

---

### 第二轮：确认资料准备情况（10分钟）

对照《腾讯文档》第三节的资料清单，逐项问：

| 问题 | 记录 |
|---|---|
| 营业执照/登记证书有电子扫描件吗？ | |
| 对公账户是哪个银行的？ | |
| 现有项目资料（介绍、照片、预算）有多少个？ | |
| 有没有机构Logo的高清图？ | |
| 有没有域名？备案了吗？ | |

---

### 第三轮：确认时间和预算（10分钟）

| 问题 | 记录 |
|---|---|
| 你们期望什么时候能用上？ | |
| 预算范围大概是多少？ | |
| 有没有技术对接人？ | |
| 后续谁负责内容（项目信息、进展更新）？ | |

---

### 第四轮：明确边界（5分钟）

**你必须说清楚的事**：

> "有几点需要提前说明：第一，我们不提供善款的金融托管——钱直接进你们对公账户，我们只负责小程序技术。第二，我们不做受助人身份真伪鉴定——证明材料是你们人工审核，系统只提供流程支持。第三，小程序上线需要微信审核，公益类审核比较严，需要你们配合提供资质文件。以上没问题吧？"

---

## 四、会议后：需求确认与签字

### 4.1 24小时内发出《需求确认书》

根据会上记录的答案，整理成一份简洁的确认书：

```markdown
# 需求确认书

项目：XX机构公益小程序
会议日期：202X年X月X日
参会人：XXX（机构方）、XXX（开发方）

## 一、确认的功能范围

### 第一期（必须做）
1. 项目广场展示 + 详情页
2. 微信支付捐款（固定档位 + 自定义）
3. 电子捐赠证书
4. ...

### 第二期（后续迭代）
1. 企业配捐
2. 月捐
3. ...

### 暂不需要
1. 发票申请
2. 数据大屏

## 二、关键业务规则
- 审核流程：两级（一线录入 → 负责人审核）
- 管理费：不提取
- 捐款档位：10/50/100/自定义
- 合作机构：需要（社区和村委会）
- 尊严支付：暂不需要
- ...

## 三、已确认的机构资料
- [ ] 营业执照（已提供）
- [ ] 项目资料（3个项目，待提供详细内容）
- [ ] Logo（待提供高清版）
- ...

## 四、时间与预算
- 期望上线：202X年X月
- 预算：X万元
- 对接人：XXX（电话：XXX）

---
确认签字：
机构方：__________  日期：__________
开发方：__________  日期：__________
```

### 4.2 发出去时的备注

> "这是根据今天会议整理的需求确认书，请核对一下功能范围和业务规则是否准确。确认没问题后我们进入设计和开发阶段。后续如果有新增或修改需求，我们会走变更流程，单独评估时间和费用。"

---

## 五、需求变更流程

开发开始后，机构可能会说"能不能加一个XX功能"。不要口头答应，要走流程：

```
机构提变更 → 你评估影响 → 输出《变更评估单》
  ├── 对排期的影响（延期X周）
  ├── 对费用的影响（增加X元）
  ├── 是否影响已开发的功能
  └── 是否可以推迟到下一期
→ 机构确认 → 签字 → 排入计划
```

**变更评估单模板**：

```markdown
# 需求变更评估

变更内容：[一句话描述]
提出人：[机构方谁提的]
提出日期：

## 影响评估
- 排期影响：
- 费用影响：
- 影响已有功能：是/否（如是，说明影响哪些）
- 建议：[本期做 / 推到下一期 / 不建议做]

## 确认
机构方签字：__________
```

## 六、常见坑与应对

| 坑 | 应对 |
|---|---|
| 机构说"你们看着做就行" | 千万别自己决定，功能确认书必须逐项确认。回头他们说"我以为有这个功能"你就没法辩解 |
| 机构说"我们要和XX平台一样" | 追问"具体哪个功能你觉得好"，拆解到功能级别而不是整体照搬 |
| 机构中途换对接人 | 确认书签字是机构公章还是个人签字？建议让机构负责人（法人/秘书长）签 |
| 问到预算时对方说"你们先报价" | 先不出具体数字，说"我需要根据今天确认的功能范围回去评估工作量，X天内给报价" |
| 对方问"能不能加个XX" | 统一回复"我先记下来，回去评估后给您回复"，不要当场答应 |
| 机构材料迟迟不提供 | 在确认书上写明"机构资料提供最晚不晚于X月X日，否则排期顺延" |
