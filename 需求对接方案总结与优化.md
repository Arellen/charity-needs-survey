# 需求对接方案 · 总结与优化

> 这套方案的核心思路：**把开放式需求讨论变成"看 → 选 → 填 → 收"的标准化流水线**。

---

## 一、方案回顾

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌──────────┐
│ 机构看       │ ──→ │ 机构选       │ ──→ │ 机构填       │ ──→ │ 你收到    │
│ 功能介绍+计划 │     │ 问卷逐项勾选  │     │ 补充说明提交  │     │ 飞书通知  │
└─────────────┘     └─────────────┘     └─────────────┘     └──────────┘
```

| 环节 | 文件 | 解决什么问题 |
|---|---|---|
| 看 | `features.html` 手机模拟图 | 机构不懂技术，"功能长什么样"一看就懂 |
| 看 | `plan.html` 分阶段计划 | 明确边界：什么先做什么后做、什么不做 |
| 填 | `survey.html` 结构化问卷 | 选择题代替开放式提问，不遗漏、不跑偏 |
| 收 | SCF → 飞书 Webhook | 提交即通知，无需手动催 |

---

## 二、当前方案的优点

### 2.1 对开发者（你）
- **零反复沟通**：机构填问卷前必须先看功能和计划，不会凭空想象功能
- **结构化输入**：问卷结果是 key-value，可以直接当作开发需求清单
- **即时通知**：飞书消息实时推送，不用等邮件/微信回复
- **历史可查**：飞书群聊天记录天然保存所有提交
- **一套模板复用到所有项目**：换个机构，改问卷选项就行

### 2.2 对机构
- **降低理解门槛**：手机模拟图比 PRD 文档好懂 100 倍
- **降低决策成本**：选择题比"你们想要什么"好回答
- **预期管理**：分阶段计划让机构知道什么能做、什么不能做、为什么
- **异步操作**：不要求双方同时在线，机构随时看、随时填

---

## 三、当前方案的不足

| # | 问题 | 严重程度 | 影响 |
|---|---|---|---|
| 1 | **提交后没持久化存储**——飞书消息是唯一记录，误删就没了 | ⚠️ 高 | 万一飞书消息被刷掉、群解散，需求记录丢失 |
| 2 | **机构提交后看不到自己填了啥**——没有确认页/摘要 | ⚠️ 中 | 万一机构说"我明明选了X"，你拿不出证据 |
| 3 | **多轮需求变更无法追踪**——机构改需求再填一次，旧版本被覆盖 | ⚠️ 中 | 不知道哪个版本是"最终版" |
| 4 | **没有材料上传入口**——机构还要另外发 Logo、资质等文件 | 🔵 低 | 增加一次额外沟通 |
| 5 | **提交后没有 CTA**——机构不知道下一步干什么 | 🔵 低 | 机构填完就忘了，等你主动联系 |
| 6 | **问卷数据是纯文本**——飞书消息不能自动生成 Excel/看板 | 🔵 低 | 每次都要手动整理 |
| 7 | **SCF 免费额度有限**——每月 100 万次调用，够用但不稳定 | 🔵 低 | 万一被攻击或被限 |

---

## 四、优化方案

### 优化 1：加一个飞书多维表格做持久化存储 ⭐⭐⭐

飞书消息容易被刷掉。在飞书里建一个**多维表格**，SCF 转发消息的同时，通过飞书 API 也写入表格一行。这样：

- 飞书消息 = 即时通知
- 多维表格 = 永久存档，可搜索、筛选、导出 Excel

**实现**：SCF 里多加一个 POST 请求到飞书表格 API。

---

### 优化 2：提交后发一份摘要给机构 ⭐⭐⭐

提交成功后，跳转到一个确认页，展示：

```
✅ 您的需求已提交（编号：20260629-001）

以下是您的选择摘要：
- 捐款功能：项目展示、在线捐款、电子证书
- 受助人管理：登记、拍照上传、两级审核
- 进阶功能：企业配捐（需要）、尊严支付（不需要）
...

我们将在 2 个工作日内联系您（138xxxx）。
如有疑问，请联系：xxx@xxx.com
```

**好处**：机构可以截图留存，防止扯皮。

---

### 优化 3：每次提交生成唯一编号，支持"改需求"场景 ⭐⭐

问卷顶部加一行提示：

> 如果您是修改之前提交的需求，请在下方输入上次提交编号（没有则留空）

新增一个字段 `revision_of`，SCF 写入表格时关联旧记录。

---

### 优化 4：问卷结果结构化写入飞书多维表格 ⭐⭐

除了发消息，SCF 还调用飞书 API 把每条问卷拆成字段写入表格：

| 提交时间 | 机构 | 填表人 | 联系方式 | 捐款功能 | 受助人管理 | 审核层级 | 企业配捐 | 尊严支付 | 期望上线 | 预算 | 编号 |
|---|---|---|---|---|---|---|---|---|---|---|
| 06-29 15:30 | XX基金会 | 张三 | 138xxx | 项目展示,在线捐款 | 登记,审核 | 两级 | 需要 | 不需要 | 3个月内 | 5-10万 | 20260629-001 |

这样你可以直接在飞书表格里排序、筛选、对比不同机构的提交。

---

### 优化 5：加一个简单的管理后台 ⭐

用飞书多维表格的「表单视图」或「看板视图」就够用，不需要额外开发。如果一定要页面，可以做个简单的 HTML 用飞书 API 读取表格数据显示。

---

### 优化 6：这套模板本身也应该复用 ⭐⭐⭐

把 `charity-miniprogram/` 整理成 **「需求对接项目模板」**：

```
~/project-template/
├── index.html          ← 改标题和入口卡片
├── features.html       ← 改模拟图（或留框架）
├── plan.html           ← 改阶段内容
├── survey.html         ← 改问卷题目
├── scf/index.py        ← 改飞书 Webhook URL 和 Secret
└── README.md           ← 使用说明
```

下次有新项目，复制一份、改问卷题目、改飞书地址，1 小时内上线。

## 五、优化优先级

| 优先级 | 优化项 | 工作量 | 效果 |
|---|---|---|---|
| **P0 立刻做** | #1 飞书多维表格存储 | 30分钟 | 解决数据丢失风险 |
| **P0 立刻做** | #2 提交后确认摘要页 | 15分钟 | 防止扯皮，给机构一个"凭证" |
| **P1 本周** | #4 问卷结果结构化写入表格 | 1小时 | 不用手动整理数据 |
| **P1 本周** | #6 整理成项目模板 | 30分钟 | 下次复用节省 80% 时间 |
| **P2 下次用之前** | #3 编号+改需求追踪 | 20分钟 | 多轮需求变更时有追溯 |
| **P2 按需** | #5 管理后台 | 不需要 | 飞书表格够用 |

## 六、最终评价

这套方案的核心价值是**把需求对接这件事标准化了**——以前需要来回微信聊、开会记笔记、事后整理 Word 发邮件，中间丢信息、歧义、返工是常态。

现在流程是：

> 发链接 → 机构自助 → 你收到结构化结果 → 直接开工

**对一个独立开发者来说，这已经足够好了。** P0 两项优化补上之后，就是一个可以放心复用的需求对接基础设施。
