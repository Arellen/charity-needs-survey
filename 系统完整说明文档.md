## 一、系统概述

### 1.1 这个系统解决什么问题

在独立开发者同时对接多个项目、多个机构时，需求沟通通常是这样：

```
发微信 → 开会 → 记笔记 → 整理Word → 来回确认 → 邮件发需求文档
```

问题是：信息丢失、版本混乱、反复确认、效率极低。

这个系统把流程标准化为：

```
发一个链接 → 机构自助查看功能介绍 → 自助填写需求问卷 → 你飞书群收到通知 + 多维表格自动存档
```

### 1.2 核心能力

| 能力 | 说明 |
|---|---|
| 📖 功能介绍展示 | 手机模拟界面，让非技术背景的机构人员直观理解功能 |
| 📅 分阶段计划 | 明确 P0/P1/P2 范围，管理预期，防止需求蔓延 |
| 📋 结构化问卷 | 选择题代替开放式提问，26 题覆盖全部功能模块 |
| 🔔 即时通知 | 机构提交后飞书群实时推送，无需等待邮件 |
| 📊 持久化存档 | 每条提交自动写入多维表格，支持搜索、筛选、导出 |
| 🔄 版本管理 | 问卷改版自动追踪，每个版本独立记录，提交数据可追溯 |
| 🏗️ 多项目复用 | 新增一个项目只需在表格加一行，30 分钟上线 |

### 1.3 当前项目

| 项目标识 | 项目名称 | 问卷链接 | 状态 |
|---|---|---|---|
| `charity` | 公益小程序 | survey.html?project=charity&ver=v1 | 使用中 |

### 2.2 数据流向

```
机构填问卷 → survey.html 收集表单数据
           → 生成格式化的消息文本
           → POST 到 SCF 云函数 (tencentscf.com)
           → SCF 查「项目目录」获取配置
           → SCF 自动维护「问卷版本」
           → SCF 计算飞书签名 → POST 飞书 Webhook → 群消息
           → SCF POST 飞书 Bitable API → 写提交记录
           → 返回成功 → 浏览器跳转 confirmed.html 确认页
```

### 2.3 关键技术点

| 技术点 | 说明 |
|---|---|
| 飞书签名算法 | HMAC-SHA256，key = `timestamp + "\n" + secret`，data = 空字符串，结果 Base64 |
| Bitable API 鉴权 | tenant_access_token（App ID + Secret 换取，2 小时有效，带缓存） |
| CORS 解决 | 浏览器 → SCF（腾讯云 Function URL 已配置跨域）→ 服务器间调用飞书 API |
| 密钥管理 | 敏感信息（App Secret、Webhook Secret）通过 SCF 环境变量注入，不写入代码 |
| 超时控制 | SCF 超时 10 秒，匹配多维表格多次 API 调用的耗时 |

## 三、飞书多维表格结构

### 3.1 表格概览

多维表格名称：「项目需求管理」

| 工作表 | 用途 | 维护方式 |
|---|---|---|
| **项目目录** | 所有项目的配置信息 | 手动维护 |
| **问卷版本** | 所有问卷版本的追踪 | 自动维护（SCF） |
| **公益小程序** | 该项目的问卷提交记录 | 自动写入（SCF） |
| **（未来项目）** | 按需新增 | 自动写入（SCF） |

### 3.2 工作表「项目目录」

一行 = 一个项目。SCF 根据 `project` 参数在此表中查找配置。

| 字段名 | 类型 | 说明 | 示例 |
|---|---|---|---|
| 项目标识 | 文本 | URL 参数用的英文短名 | `charity` |
| 项目名称 | 文本 | 显示名称 | `公益小程序` |
| 项目简介 | 文本 | 一句话描述 | |
| 项目状态 | 单选 | 需求对接中/开发中/已上线/已完成/暂停 | |
| webhook_url | 文本 | 飞书群机器人的 Webhook 地址 | `https://open.feishu.cn/...` |
| webhook_secret | 文本 | 飞书群机器人的签名密钥 | `RbVRjrUp...` |
| 提交表ID | 文本 | 该项目提交数据写到哪个工作表的 table_id | `tblvS2hZ...` |
| 创建时间 | 日期 | | |
| 备注 | 文本 | | |

### 3.3 工作表「问卷版本」

一行 = 一个问卷版本。SCF 根据 `project` + `ver` 参数自动创建和维护。

| 字段名 | 类型 | 说明 | 示例 |
|---|---|---|---|
| 所属项目 | 文本 | 项目名称（注意：是「公益小程序」不是 `charity`） | `公益小程序` |
| 版本号 | 文本 | URL 参数用的版本号 | `v1` `v2` `v3` |
| 问卷名称 | 文本 | 该版本问卷的名称 | `需求确认问卷` |
| 问卷链接 | 文本 | 该版本问卷的完整 URL | `https://...?project=charity&ver=v2` |
| 状态 | 单选 | 使用中/已停用/草稿 | |
| 对接机构 | 文本 | 如果该版本专门发给某个机构，可注明 | |
| 备注 | 文本 | 记录该版本改动内容 | |

自动创建逻辑：
- 当收到 `project=X&ver=Y` 的提交，检查表中是否已有 `(X, Y)` 行
- 不存在 → 自动创建，状态设为「使用中」，链接自动填入
- 已存在 → 不重复创建

### 3.4 工作表「公益小程序」

一行 = 一次问卷提交。SCF 每次收到提交后自动写入。

| 字段名 | 类型 | 说明 |
|---|---|---|
| 提交编号 | 文本 | RQ+年月日-时分，如 `RQ20260630-1530` |
| 提交时间 | 日期 | 自动填充 |
| 机构名称 | 文本 | Q1 |
| 填表人 | 文本 | Q2 |
| 联系方式 | 文本 | Q3 |
| 现有平台 | 文本 | Q4 |
| 当前管理方式 | 文本 | Q5 |
| 捐款功能 | 文本 | Q6（多选用顿号分隔） |
| 捐款档位 | 文本 | Q7 |
| 允许匿名 | 文本 | Q8 |
| 场景化入口 | 文本 | Q9 |
| 捐款反馈 | 文本 | Q10 |
| 受助人功能 | 文本 | Q11 |
| 登记字段 | 文本 | Q12 |
| 证明材料 | 文本 | Q13 |
| 审核层级 | 文本 | Q14 |
| 合作机构录入 | 文本 | Q15 |
| 项目管理功能 | 文本 | Q16 |
| 管理费 | 文本 | Q17 |
| 尊严支付 | 文本 | Q18 |
| 企业配捐 | 文本 | Q19 |
| 月捐 | 文本 | Q20 |
| 数据大屏 | 文本 | Q21 |
| 期望上线 | 文本 | Q22 |
| 预算范围 | 文本 | Q23 |

## 五、前端页面说明

### 5.1 页面清单

| 文件 | URL 示例 | 用途 |
|---|---|---|
| `index.html` | `/` | 首页，三个入口卡片 |
| `features.html` | `/features.html` | 功能介绍，含手机模拟界面 |
| `plan.html` | `/plan.html` | 分阶段计划（P0/P1/P2） |
| `survey.html` | `/survey.html?project=charity&ver=v1` | 需求问卷，带项目和版本参数 |
| `confirmed.html` | 提交成功后自动跳转 | 确认页，显示编号和下一步指引 |

### 5.2 URL 参数说明

`survey.html` 接受两个 URL 参数：

| 参数 | 必填 | 默认值 | 说明 |
|---|---|---|---|
| `project` | 否 | `charity` | 项目标识，对应「项目目录」表中的 `项目标识` 字段 |
| `ver` | 否 | `v1` | 问卷版本号，对应「问卷版本」表中的 `版本号` 字段 |

示例：
- 公益小程序 v1：`?project=charity&ver=v1`
- 公益小程序 v2：`?project=charity&ver=v2`
- 未来 XX 项目 v1：`?project=xxproject&ver=v1`

### 5.3 提交逻辑

```
点击提交
  → 收集表单数据 → 格式化为飞书消息文本
  → POST 到 SCF { text, project, ver }
  → SCF 处理 → 返回 { code: 0 }
  → 存 sessionStorage → 跳转 confirmed.html
  → confirmed.html 读取数据 → 展示确认页

如果 SCF 不可达：
  → 页面显示错误信息 + 完整表单数据
  → 机构可截图发给对接人
```

## 六、日常操作指南

### 6.1 给机构发问卷

1. 确认「项目目录」和「问卷版本」表数据正确
2. 把 URL 发给机构：`https://arellen.github.io/charity-needs-survey/survey.html?project=charity&ver=v1`
3. 机构打开链接 → 查看功能介绍 → 填写问卷 → 提交
4. 你的飞书群收到消息 + 多维表格多一行

### 6.2 查看历史提交

1. 打开飞书多维表格「项目需求管理」
2. 点进对应项目的工作表
3. 可进行：搜索、筛选、排序、导出 Excel

### 6.3 修改问卷内容

1. 编辑 `survey.html` 中的问卷题目
2. 提交到 GitHub → 自动部署到 GitHub Pages
3. 使用新版本号 `ver=v2` 发给机构
4. 无需改 SCF 代码，无需改多维表格
5. 第一次 v2 提交时，「问卷版本」表自动创建 v2 行

### 6.4 修改问卷后重新发给同一机构

如果机构 v1 填过了，需要填 v2：
1. 发新链接：`?project=charity&ver=v2`
2. 多维表格里会有两条记录（v1 和 v2），通过「版本标记」字段区分

## 七、新增项目操作流程

新增一个项目（如「XX助学项目」，标识 `edu2026`）：

### 步骤 1：飞书多维表格

- 在「项目需求管理」中新建一张工作表，命名「XX助学项目」
- 字段结构复制「公益小程序」那张表（可右键复制工作表）
- 记录下新工作表的 **table_id**

### 步骤 2：「项目目录」表加一行

| 项目标识 | 项目名称 | webhook_url | webhook_secret | 提交表ID |
|---|---|---|---|---|
| `edu2026` | XX助学项目 | `https://...` | `xxx` | `tblxxxxx` |

### 步骤 3：建飞书群 + 机器人

- 新建飞书群（如「XX助学-需求对接」）
- 添加自定义机器人 → 复制 Webhook URL 和 Secret
- 填入「项目目录」表对应行

### 步骤 4：SCF 环境变量

添加两条：
```
EDU2026_WEBHOOK_URL = xxx
EDU2026_WEBHOOK_SECRET = xxx
EDU2026_TABLE_ID = tblxxxxx
```

### 步骤 5：准备问卷

- 复制 `survey.html` → 按项目需要修改题目
- 完整问卷链接：`/survey.html?project=edu2026&ver=v1`

### 步骤 6：测试

```
浏览器打开 ?project=edu2026&ver=v1 → 填问卷 → 提交
检查：① 飞书群收到消息 ② 多维表格「XX助学项目」多一行 ③「问卷版本」自动创建
```

> 全程无需改 SCF 代码，只需在表格和环境变量里加配置。

## 八、故障排查

### 8.1 提交后飞书群没收到消息

| 可能原因 | 检查方法 |
|---|---|
| SCF 函数挂了 | 腾讯云 SCF → 日志查询 |
| Webhook URL 或 Secret 错误 | 检查「项目目录」表对应行的值 |
| 环境变量没配 | SCF → 函数配置 → 环境变量 |
| 飞书机器人被移除 | 检查飞书群机器人是否还在 |

### 8.2 多维表格没写入

| 可能原因 | 检查方法 |
|---|---|
| App ID/Secret 错误 | SCF 环境变量 `APP_SECRET` 是否正确 |
| 应用没权限 | SCF 日志中是否有 `403 Forbidden` |
| 应用没发布 | 飞书开放平台 → 版本管理与发布 |
| 表格没授权给应用 | 多维表格 → 更多 → 添加应用 |
| 字段类型冲突 | SCF 日志中是否有 `TextFieldConvFail` 等错误 |
| 超时 | SCF 超时时间是否 ≥ 10 秒 |

### 8.3 提交后页面显示空白

| 可能原因 | 检查方法 |
|---|---|
| SCF 返回异常 | 浏览器 F12 → Network → 查看 SCF 响应 |
| sessionStorage 数据丢失 | 确认页代码是否正常加载 |

### 8.4 「问卷版本」表重复创建

| 可能原因 | 检查方法 |
|---|---|
| 项目名称不一致 | `ensure_version` 中的项目名和表中已有行的项目名是否一致 |
